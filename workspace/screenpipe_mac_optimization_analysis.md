# Анализ производительности Screenpipe на macOS и возможности оптимизации

Этот документ содержит глубокий технический анализ архитектуры Screenpipe, разбор причин высокого потребления памяти и процессора на macOS, обзор готовых альтернатив, а также оценку целесообразности переписывания проекта на **Swift**, **Go** или **C++**.

---

## 1. Анализ текущей архитектуры и узких мест Screenpipe на macOS

Хотя Screenpipe позиционируется как высокопроизводительный инструмент на Rust, при работе на macOS он сталкивается с рядом фундаментальных ограничений и накладных расходов.

### А. CPU-bound обработка изображений в Rust
В файле [`apple.rs`](file:///Users/nikshilov/.gemini/antigravity/scratch/screenpipe/crates/screenpipe-screen/src/apple.rs#L119) можно увидеть следующую строку:
```rust
let rgb = image.grayscale().to_luma8();
let raw_data = rgb.as_raw();
```
* **Проблема:** Перед передачей кадра в Apple Vision OCR, Rust-сервер переводит полноразмерный снимок экрана (который на Retina-дисплеях имеет гигантское разрешение) в градиент серого и формат Luma8 **на процессоре (CPU)**.
* **Эффект:** Обработка миллионов пикселей на CPU каждые 1-2 секунды вызывает постоянный нагрев процессора и повышенное энергопотребление.
* **Как должно быть в нативной среде:** Нативный macOS-фреймворк `ScreenCaptureKit` предоставляет видеокадры напрямую в виде `CVPixelBuffer` (хранящихся в памяти GPU). Их можно сразу передать в Vision API (`VNImageRequestHandler`) **без единого копирования байтов и без участия CPU**.

### Б. Утечки памяти на стыке Rust ⟷ Objective-C (FFI)
В macOS управление памятью системных фреймворков завязано на `NSAutoreleasePool`. В Rust-коде автоматического освобождения таких объектов не происходит. 
* В истории коммитов Screenpipe (например, в файле [`monitor.rs`](file:///Users/nikshilov/.gemini/antigravity/scratch/screenpipe/crates/screenpipe-screen/src/monitor.rs#L1103-L1200)) зафиксирован критический баг: утечка памяти до **13.2 ГБ RAM** за 48 часов работы. Причиной было отсутствие вызова `ar_pool` во второстепенных потоках при опросе мониторов.
* Любая сторонняя библиотека-биндер (в данном случае `cidre` для связи Rust с Apple API) несет в себе риск пропустить подобные утечки. В нативном Swift ARC (Automatic Reference Counting) контролирует это на уровне компилятора, а синтаксис `@autoreleasepool` применяется нативно.

### В. Накладные расходы на интерфейс (Tauri/Electron)
Screenpipe использует Tauri (или Electron) для отображения таймлайна.
* Tauri использует системный WebKit (`WKWebView`). Хотя он намного легче Chromium в Electron, он всё равно запускает отдельный процесс рендеринга JS/CSS, который потребляет **150–300 МБ RAM** в фоновом режиме просто для удержания окна в памяти.
* Нативное приложение на **SwiftUI** потребляет всего **15–40 МБ RAM** на весь интерфейс и иконку в меню-баре.

### Г. Тяжелый Whisper для аудио-транскрипции
Локальный запуск Whisper (даже через оптимизированный `whisper.cpp`) требует удержания в памяти весов модели (от 70 МБ для Tiny до 1.5 ГБ для Large) и периодического сильного прогрева GPU.

## 2. Существующие нативные альтернативы на macOS

Если вы ищете что-то, что работает плавно, не нагружает систему и уже написано на Swift, обратите внимание на следующие проекты:

### 1. **TimeScroll** (Рекомендуется к изучению)
* **Репозиторий:** [XInTheDark/TimeScrollApp](https://github.com/XInTheDark/TimeScrollApp)
* **Язык:** 100% Swift / SwiftUI
* **В чём суть:** Это полностью локальный аналог Rewind/Recall для macOS.
* **Почему он быстрый:**
  1. **Bypass OCR via Accessibility API:** Вместо того чтобы постоянно делать скриншоты и прогонять их через OCR (что нагружает процессор и видеокарту), TimeScroll извлекает текст напрямую из системного дерева доступности (Accessibility API) активного приложения. OCR включается только как редкий фоллбек.
  2. **HEVC + De-duplication:** Скриншоты сжимаются нативными аппаратными кодеками macOS (HEVC) и сохраняются только при изменении экрана, занимая минимум места.
  3. **Нативный UI:** Потребляет минимум ресурсов, бесшовно интегрирован в систему.

### 2. **BetterCapture**
* **Репозиторий:** [jsattler/BetterCapture](https://github.com/jsattler/BetterCapture)
* **Язык:** Swift / SwiftUI
* **В чём суть:** Нативный рекордер экрана на базе `ScreenCaptureKit`. Отличная кодовая база для понимания того, как захватывать экран с нулевыми потерями производительности и аппаратным кодированием в ProRes/HEVC.

## 3. Сравнение языков для потенциального переписывания

Если вы задумываетесь о написании собственной утилиты захвата или переносе логики, вот честный разбор вариантов:

| Критерий | Go | C++ | Swift (Наш выбор) |
| :--- | :--- | :--- | :--- |
| **Интеграция с macOS API** | ❌ Ужасно. Требует `cgo`, постоянные конвертации типов, медленные переходы границ C-Go. | ⚠️ Сложно. Требует Objective-C++ (`.mm` файлов) и ручного управления указателями. |  Отлично. Полная нативная поддержка всех Apple фреймворков без переходников. |
| **Управление памятью** | ⚠️ GC (Garbage Collector) может вызывать периодические скачки памяти и лаги. | ⚠️ Ручное. Высокий риск утечек в долгоживущем фоновом процессе. |  ARC (Automatic Reference Counting). Предсказуемое, минимальное потребление RAM. |
| **Аппаратное ускорение** | ❌ Нет прямого доступа к CoreVideo, Metal и Neural Engine без C-обёрток. |  Отличный низкоуровневый доступ к GPU. |  Прямой доступ к CoreImage, Accelerate, Vision и CoreML. |
| **Кроссплатформенность** |  Отличная. |  Хорошая. | ❌ Отсутствует (только Apple-экосистема). |

### Почему Go — плохой выбор для этой задачи?
Фоновый захват экрана и звука требует теснейшей интеграции с системными вызовами macOS. В Go вам придется писать 80% кода на Objective-C/C через `cgo`. Это сведет на нет все преимущества Go, усложнит сборку и создаст огромный оверхед по производительности на границе Go-кода и системных C-библиотек.

### Почему C++ — избыточен и опасен?
C++ даст максимальную скорость, но цена ошибки (утечка памяти или segmentation fault) в приложении, работающем 24/7, слишком высока. Кроме того, современные API от Apple (вроде ScreenCaptureKit или нового Apple Intelligence) проектируются с упором на Swift-конкурентность (`async/await`) и тяжело импортируются в чистый C++.

## 4. Почему Swift — идеальный стек для идеальной «Таймлайны»

Если переписывать Screenpipe (или создавать аналог) под Mac с целью сделать его "невидимым" для батареи и памяти, **Swift — безальтернативный победитель**:

1. **Zero-Copy конвейер захвата:**
   * Кадр захватывается через `ScreenCaptureKit` напрямую в память видеокарты (`CVPixelBuffer`).
   * Без копирования в оперативную память и без участия CPU, этот буфер передается в Vision API для распознавания текста.
   * Текст распознается на сопроцессоре Neural Engine (ANE), оставляя CPU абсолютно холодным.
2. **Использование Apple Speech Framework:**
   * Вместо компиляции тяжелого Whisper, Swift-приложение может использовать встроенный в macOS `SFSpeechRecognizer`.
   * Он использует локальные языковые модели Apple, оптимизированные под конкретный чип Apple Silicon, обновляется системой и не занимает память вашего приложения.
3. **Нативный SwiftUI интерфейс:**
   * Окно поиска, графики активности и красивый стеклянный (Glassmorphism) таймлайн будут работать со скоростью 120 кадров в секунду (ProMotion) силами родного графического движка Metal, расходуя всего 25 МБ RAM.
4. **Нативное хранилище:**
   * Использование CoreData или нативного SQLite (через GRDB.swift) гарантирует быстрые транзакции и оптимизированное кэширование без накладных расходов на сетевые протоколы.

## 5. Выводы и рекомендации

* **Стоит ли переписывать?** 
  Если ваша цель — получить **идеально оптимизированный инструмент исключительно для macOS**, который не греет ноутбук и ест 50 МБ памяти вместо 4 ГБ — **однозначно да, на Swift**.
* **С чего начать?**
  Прежде чем писать с нуля, обязательно посмотрите и скомпилируйте [TimeScroll](https://github.com/XInTheDark/TimeScrollApp). Этот проект уже решает 90% ваших задач на чистом Swift, используя системное дерево доступности для экономии энергии. Возможно, вам будет проще дописать нужные функции (например, запись звука) туда, чем создавать всё заново.
* **Как оптимизировать текущий Screenpipe без переписывания?**
  1. Зайдите в настройки и ограничьте FPS захвата экрана до 0.5 (1 кадр в 2 секунды).
  2. Переключите OCR на нативный Apple (вместо Tesseract), если это не сделано по умолчанию.
  3. Закрывайте UI-окно Screenpipe, когда им не пользуетесь, чтобы выгрузить WebKit-процессы Tauri.
  4. Отключите аудио-захват, если вам не нужна транскрипция голоса (Whisper — самый тяжелый компонент).
