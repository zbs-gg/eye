# Архитектурная стратегия перехода на нативный Swift (Pure Swift Rewrite)

Этот документ описывает стратегию переноса функционала Screenpipe (REST API, коннекторы, плагины и MCP-сервер) на 100% нативный стек macOS с использованием **Swift**, а также предлагает многоагентный пайплайн разработки для реализации этой масштабной задачи.

---

## 1. Нативная реализация API и коннекторов на Swift

Перенос сетевого слоя и расширяемости на Swift не только возможен, но и сделает приложение гораздо более легким, так как отпадет необходимость тянуть за собой тяжелые рантаймы.

### А. Локальный REST API (Вместо Rust Axum)
Для организации локального HTTP-сервера на macOS есть два прекрасных современных решения, полностью построенных на структурированной конкурентности (`async/await`):
1. **Hummingbird 2.0:** Легковесный, модульный веб-фреймворк, разработанный сообществом Swift на базе высокопроизводительного `SwiftNIO`. Он отлично подходит для быстрого роутинга и JSON-сериализации.
2. **FlyingFox:** Чрезвычайно минималистичный веб-сервер, написанный на чистом Swift без тяжелых зависимостей. Идеален для встраивания прямо в macOS Menu Bar приложение.

### Б. База данных (Вместо Rust SQLx)
* **Решение:** **GRDB.swift** (самая быстрая и продвинутая библиотека для работы с SQLite в экосистеме Apple).
* **Преимущества:** Поддерживает многопоточный режим WAL (Write-Ahead Logging), полнотекстовый поиск (FTS5) для быстрого поиска по распознанному тексту и реактивные запросы (UI и API автоматически узнают о новых записях без постоянного опрашивания БД).

### В. Коннекторы и плагины (Pipes)
В Screenpipe плагины работают на JS/WASM. В Swift это можно реализовать невероятно элегантно и без оверхеда:
* **JavaScriptCore (JSC):** Это движок Safari, встроенный прямо в macOS на уровне операционной системы. Мы можем выполнять кастомные JavaScript-скрипты пользователей для обработки данных из БД. JSC работает молниеносно, потребляет около **1–2 МБ памяти** (в отличие от Node.js или V8) и интегрируется в Swift в 5 строк кода.
* **Apple Shortcuts (Команды):** Можно предоставить нативные действия (App Intents) для системного приложения «Команды». Это позволит пользователям автоматизировать обработку истории экрана без написания кода, связывая приложение с Notion, Telegram и т.д.

### Г. MCP (Model Context Protocol) Сервер
* **Решение:** Использование **[официального Swift SDK для MCP](https://github.com/modelcontextprotocol/swift-sdk)**, созданного разработчиками протокола.
* **Как это работает:** Swift-приложение запускает легковесный stdin/stdout или SSE-транспорт, который общается с LLM-клиентами (Claude Desktop, Cursor, VS Code). Весь контекст вашей экранной истории становится доступен вашему ИИ-помощнику напрямую через высокопроизводительные запросы к GRDB.

## 2. Пайплайн разработки: Как правильно переписать проект

Для реализации такого проекта силами ИИ-агентов необходима четкая декомпозиция задач и использование специализированных агентов. Один агент не сможет удержать в контексте и архитектуру SCK (видео-потоки), и обработку аудио, и оптимизацию базы данных.

### Шаг 1: Фаза архитектурного планирования (System Architect Agent)
* **Задача:** Проектирование схемы данных в `GRDB.swift` и Core Data, описание протоколов для zero-copy буферов видео (`CVPixelBuffer` ⟷ `VNImageRequestHandler`).
* **Результат:** Swift Package Manager (SPM) структура проекта с четко разделенными таргетами: `CoreCapture`, `Database`, `APIServer`, `MCPServer`, `AppUI`.

### Шаг 2: Создание ядра захвата (Core Capture & ML Agent)
* **Задача:** Настройка `ScreenCaptureKit` для умного стриминга окон/экранов, интеграция системного `SFSpeechRecognizer` для звука и `Vision` API для OCR.
* **Результат:** Автономный модуль, который пишет данные в WAL-базу с частотой, зависящей от активности пользователя, расходуя не более 1–2% CPU.

### Шаг 3: Написание сетевого слоя и MCP (Database, API & MCP Agent)
* **Задача:** Развертывание `FlyingFox` сервера для REST API, интеграция официального `swift-sdk` для MCP.
* **Результат:** Локальный REST API порт и готовый к подключению MCP-сервер, транслирующий данные из локальной SQLite.

### Шаг 4: Разработка нативного UI (SwiftUI UX Agent)
* **Задача:** Создание потрясающего интерфейса: иконка в Menu Bar, панель поиска «Spotlight-style» с размытием фона (Glassmorphism) и плавный интерактивный таймлайн на SwiftUI.
* **Результат:** Легковесный графический интерфейс, потребляющий до 35 МБ RAM.

### Шаг 5: Профилирование и оптимизация батареи (Performance QA Agent)
* **Задача:** Анализ утечек памяти через Xcode Instruments (Leaks, Allocations), оптимизация энергопотребления (Energy Log), отладка фонового режима сна macOS.

## 3. Использование суперсилы MCP (MCP Superpower) в разработке

В процессе написания этого проекта мы можем использовать возможности MCP для ускорения разработки:
1. **Написание кода с использованием лучших практик:** Мы можем подключить MCP-серверы, которые знают все последние изменения в `ScreenCaptureKit` и Swift 6 Concurrency (так как Apple API меняется быстро, и веса моделей ИИ часто содержат устаревшие паттерны).
2. **Интерактивная отладка:** С помощью MCP-инструментов разработчик может компилировать куски Swift-кода в реальном времени, тестировать производительность захвата кадров и сразу анализировать логи утечек памяти.
3. **Автоматическая генерация тестов:** Использование MCP для запуска локальных тестов производительности (performance tests) на macOS, чтобы убедиться, что новый Swift-код действительно потребляет в 10 раз меньше ресурсов, чем старый Rust/Tauri код.
